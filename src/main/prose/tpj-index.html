Tutorial: Prepare for Java Exercises

After completing this tutorial, you are familiar with the structure of the upcoming Java classroom exercises. You know how to perform the required development procedures in your preferred development environment.

This tutorial comprises three phases:
1. Run the JAR
You execute the provided, runnable JAR to understand what it is supposed to do, and what is wrong with it.
2. Study the Sources
You look at the provided JavaDocs and source files to find your way through the code.
3. Fix the Problems
You fix the source code and build your own, working JAR from it.
The recipe outlines the steps to follow when working on a Java classroom exercise. You practice most of these steps in the tutorial. Last but not least, there is some information about the upcoming exercises. Reading it before the first classroom exercise gives you a head start.

Run the JAR

tutorial-faulty.jar
The program to test and fix. Usage instructions:
  java -jar tutorial-faulty.jar
Sources are in the jar, docs here.

First verify that a Java Virtual Machine (JVM) or Java Development Kit (JDK) is available on your computer. Open a command line and enter:

java -version

Version 1.7 or later is needed. If the java command is not found, you have to investigate why. Most likely it is just missing from the search path. Locate an installed JVM or JDK and add its bin directory to the PATH environment variable, or use the full path to run java and related utilities. If there is neither JVM nor JDK, install a JDK. You'll need it later anyway. How to do that depends on your operating system and the JDK you want to use.

Now download the file tutorial-faulty.jar to your machine. Change to the directory where you saved it and enter:

java -jar tutorial-faulty.jar

This tells java to run the program contained in the JAR file. Without further arguments, that program just prints usage instructions. Take a minute to read those.

Seriously, read them.

So, the program understands several different commands. Use latin to check whether your environment is ready to display characters from the ISO Latin 1 character set.

java -jar tutorial-faulty.jar latin

If the output contains umlauts, you're all set. If it contains garbage, or question marks in unexpected places, you need to do something. The default character encoding of the JVM does not match that of your command line. This is a common problem with cmd.exe on Windows. You can use a different command line, for example the PowerShell, or figure out how to fix the problem.

Test the Commands

The downloaded JAR supports three more commands besides latin. They cover the technical content of this tutorial: IP addresses and hostnames. Let's try them one after another, starting with local.

java -jar tutorial-faulty.jar local
Network Interface 'wlan0' fe80:0:0:0:fc5a:73ee:fe85:733c%3 10.223.123.161 Network Interface 'lo' 0:0:0:0:0:0:0:1%1 ip6-localhost 127.0.0.1 localhost

The output depends on the network configuration and available connections. In the example, the machine is connected to a WLAN, 'wlan0'. The local interface, 'lo', is always present. It enables programs that run on the same machine to communicate with eachother, even if there is no actual network connection. The machine from the example has both an IPv4 and an IPv6 address on either network interface. Does yours?

The remaining two commands expect an argument, which can be a hostname or an IP address. The usage instructions only mention hostnames, but all operating systems are smart enough to handle IP addresses as well. Let's try both commands with 'localhost' as the argument.

java -jar tutorial-faulty.jar islocal localhost
Exception in thread "main" pityoulish.outtake.Missing: bind a socket to a hostname at ...
java -jar tutorial-faulty.jar lookup localhost
Looking up 'localhost'... Exception in thread "main" pityoulish.outtake.Missing: look up the InetAddress(es) of a hostname at ...

Dang! Both commands are broken. Well, there had to be a reason why the JAR is called 'faulty'. The class of the exception, pityoulish.outtake.Missing, indicates that something was intentionally left out of the program. And you are expected to fill it in. If you see a different exception, there is a problem. Read the exception message. At best, it's just a typo in the command line.

Study the Sources

Before you proceed, rename the downloaded JAR to tutorial.jar or some other name without negative connotation. While it is faulty now, it won't be when you're done with it. Renaming the file supports a positive mindset right from the beginning.

List the contents of the JAR file. Use the jar utility of a JDK, or unzip, or any other archive utility that is compatible with the zip format. The example output below shows only a subset of the files.

jar -tf tutorial.jar
META-INF/MANIFEST.MF pityoulish/cmdline/ArgsInterpreter.java ... pityoulish/outtake/Missing.java pityoulish/outtake/package.html pityoulish/tutorial/Main.java pityoulish/tutorial/TutorialCommandHandler.java pityoulish/cmdline/ArgsInterpreter.class ... pityoulish/outtake/Missing.class pityoulish/tutorial/Main.class ... pityoulish/tutorial/TutorialCommandHandler.class

On first glance, it seems as if classes are listed twice. Indeed, a closer look reveals that the JAR contains source code (.java) along with compiled classes (.class). The presence of 'package.html' files suggests that there might even be meaningful JavaDocs. It is always possible to generate JavaDocs from source code. But if the source code doesn't contain special JavaDoc comments, the generated docs are not necessarily useful. Alas, the mere presence of JavaDoc comments also doesn't ensure that they contain anything useful. Fortunately, this is a tutorial with well-documented source code.

If you plan to use an Integrated Development Environment (IDE) for coding, now is a good time to import tutorial.jar as a new project. There are many IDEs that support Java development, for example IntelliJ, Eclipse, Visual Studio Code, or NetBeans. However, you don't need an IDE to complete this tutorial, or the upcoming programming exercises. Just use the archiver from above to extract the files from the JAR, and your favorite editor to browse them. Make sure to maintain the directory structure when extracting though.

JavaDoc Walk-Through

Before you dive into the source code and get lost in its intricacies, you should get a high-level overview. Modern IDEs make it very easy to access context-specific information. For example, they can show the JavaDocs for a method as hover help, right there where the method is being called. At the same time, they beguile users into neglecting the bigger context. Why bother with the class- or package-level JavaDocs when the method-level is readily available? Never mind that the former might point out other classes or methods that would serve your purposes better, or provide valuable information that applies to all the methods of a class. To avoid this pitfall, don't go into your IDE or editor just yet.

Open the JavaDocs in a new browser tab or window, so you can easily switch between them and this description. Check the list of packages. There are only two, pityoulish.cmdline and pityoulish.tutorial. In the main content frame, click on the first one. Now you see a list of interfaces and classes in that package. Glimpse at the names. They are about commands, handling, and arguments. The package description begins with "Utility classes". This indicates that the package does not contain the main functionality of the program, but supporting features. Are you interested in learning how the command-line arguments are interpreted? If so, write it down on your to-do list for later. Right now, the support functionality is not your concern.

Next, access the JavaDocs for the pityoulish.tutorial package. There is a class Main, described as the main entry point. Now you know where to start. There is an enum with four constants, named like the four commands which tutorial.jar understands. And there is a handler class with methods named like the four commands. Could handleLatinCmd be the method which prints the umlauts for the 'latin' command? Yes, the description says so. Without even looking at the source code, you've learned where the important things are happening.

Source Code Walk-Through

Now it is time to browse the source code. If you didn't know already from the JavaDocs where to start, you would have to look into the file META-INF/MANIFEST.MF. It defines a property 'Main-Class', which tells java -jar about the entry point of the program. Unsurprisingly, the value is pityoulish.tutorial.Main. Open the source file for that class in your IDE or editor and find the method main.

The main method initializes an arg(ument)s interpreter, calls it with the command-line arguments, and is done. If you hadn't looked at the JavaDocs first, you would now probably follow the program execution and look into the args interpreter. However, you already know that it comes from a utility package, and that the commands are implemented in TutorialCommandHandler. An instance of that class is given to the args interpreter as a constructor argument. The other constructor argument is some kind of 'None', so probably not relevant here. The natural conclusion is that the args interpreter will call the command handler. Let's go directly to the latter, without wasting time on the former.

Looking at TutorialCommandHandler, you see the enumeration of supported commands, a method that returns usage instructions, and a method with a switch statement on the enum. Each case of that statement calls a handler method for the respective command. Whatever else the args interpreter does, it calls here to execute a command. That's all you need to know about it to get started.

Word of Advice Badly structured code invalidates natural conclusions. When you are diving into any codebase of significant size, it is impossible to understand every part of it to the very last detail. In order to find the code of interest within a reasonable time, you absolutely must make assumptions about other parts of the code. In a well-structured codebase, classes and modules behave roughly as you would expect from their names or descriptions. This is called the Principle of Least Astonishment (POLA). It greatly simplifies the task of getting to know a codebase and reduces the time you need to become productive.
In ill-structured code, such assumptions will not hold. Classes or modules do things you wouldn't expect, and things you do expect are not done where you expect them. You'll spend a lot of time searching for why something is happening, until you find the cause hidden away in an inconspicuous module. Or you'll read through some code and see that it is doing completely unrelated things as well. These moments of realization are called WTFs. In fact, WTFs per minute are a measurement of code quality. For a regular stream of bad examples, see The Daily WTF.

Fix the Problems

There are two broken commands to fix in this tutorial. The following instructions for the first are very detailed. Those for the second remain brief, so you can apply what you learned. Finally, you should clean up.

Fix the islocal Command

Call the command again, to have a closer look at the error message and the stack trace. Details such as line numbers probably differ from the example output. Work with what you see in your command-line terminal, because that matches with the source code in your JAR.

java -jar tutorial.jar islocal localhost
Exception in thread "main" pityoulish.outtake.Missing: bind a socket to a hostname at pityoulish.outtake.Missing.here(Missing.java:31) at pityoulish.tutorial.TutorialCommandHandler.handleIsLocalCmd(TutorialCommandHandler.java:159) at pityoulish.tutorial.TutorialCommandHandler.handleCommand(TutorialCommandHandler.java:92) at pityoulish.tutorial.TutorialCommandHandler.handleCommand(TutorialCommandHandler.java:26) at pityoulish.cmdline.CommandHandlerBase.handle(CommandHandlerBase.java:76) at pityoulish.cmdline.ArgsInterpreter.handle(ArgsInterpreter.java:88) at pityoulish.tutorial.Main.main(Main.java:27)
Interpret the Stack Trace

The first line reveals the class of the exception and the exception message. The class often indicates the type of problem, for example a NullPointerException or an IOException. In this case, some code is missing; you already know that. The message tells you what the missing code is supposed to do. We'll get to that later, once you've found the place where it is missing.

The lines starting with "at" are the call stack. The program execution started at main, which is in the last line. main called ArgsInterpreter.handle, which called CommandHandlerBase.handle, which called something else, and so on, right up to Missing.here at the top of the stack. And there, the exception was thrown. In parenthesis, you see the source file name 'Missing.java' and a line number. Open the file and find the line number indicated by your stack trace. If you executed the command from within an IDE, you probably just have to click a link in the output. Now what is wrong in here?

Well, nothing is wrong in Missing.here. It is a helper method meant to throw an exception, and so it does. The caller must have found that something is wrong and raised an exception by invoking the helper. You need to find the place from which it is being called. Luckily, the second line of the stack trace tells you exactly that. Load the file 'TutorialCommandHandler.java' and have a look at method handleIsLocalCmd, especially around the line number from your stack trace. There you'll see the following statements:

Missing.here("bind a socket to a hostname"); Missing.pretend(IOException.class);
Analyze the Problem

Before you can fix anything, you have to understand what is going on in that part of the code. The references to class Missing are exactly where you'll have to put in your own code. The message, 'bind a socket to a hostname', tells you what is missing. But unless you are familiar with the socket API, this message is unclear. The call to Missing.pretend indicates that the missing code may throw an IOException.

When a method is so small as handleIsLocalCmd, you should try to understand it fully. There is a description, which you can read in the source code as well as in the JavaDocs. The method is expected to return 0 if the argument is the local host, or -1 if it is not. In the method body, there is some error handling code. If the missing code runs through, the method returns 0. If it throws an IOException, the method returns -1. Before that, the string argument is converted into an InetSocketAddress. Let's summarize:

  1. You have a hostname, already converted to InetSocketAddress.
  2. The missing code should run through if the hostname is local.
  3. The missing code should throw an IOException if the hostname is not local.
  4. The missing code should 'bind' a 'socket' to the hostname.

Now it is time to study the standard Java APIs. There are JavaDocs available: online, for download, or maybe already installed with your JDK or IDE. Pick whichever is most convenient for you. The InetSocketAddress is a place to start. Look for the import statements at the beginning of 'TutorialCommandHandler.java'. From which package is that class? What other classes are imported from there? Anything related to 'bind' or 'socket'? Bingo, there's an import of java.net.Socket. Search and browse the JavaDocs for these two classes. Is there something about 'bind'? Can you do that with an InetSocketAddress? Does it make a difference whether the hostname is local?

When you have found and assimilated that information, put in the missing code. Delete the calls to class Missing. Create an unconnected socket. Bind that socket to the hostname. That's it. Maybe. To be sure, you need to test your changes.

Build the JAR

If you are using an IDE, it compiles modified source files automatically. IDEs also provide an option to export your project as a JAR file. But you have to check whether that JAR file can be run with java -jar. If not, look for options to specify the property 'Main-Class' of the manifest. If all else fails, you can still run your code by spelling out the classname. It's less convenient, but works just as well. These two commands are equivalent, both print the usage instructions:

java -jar tutorial.jar
java -cp tutorial.jar pityoulish.tutorial.Main

Without an IDE, you have to call the compiler and build the JAR explicitly. Assuming that the current directory is where you extracted the JAR, use the following commands:

javac pityoulish/tutorial/TutorialCommandHandler.java
jar -cfe tutorial.jar pityoulish.tutorial.Main pityoulish/

The first command calls the Java compiler, javac. The argument is the source file to compile. You can specify more than one, or use wildcards to select all the source files you modify. Without other arguments, the compiler puts the compiled .class files into the same directory as the source files. The second command, jar, creates ('c') a new JAR file ('f') named 'tutorial.jar' and sets the entry point ('e') to class pityoulish.tutorial.Main. The JAR file contains everything from directory 'pityoulish', including all subdirectories.

Test the Fix

Finally, you're ready to test the islocal command with your changes. Try it with some IP addresses and hostnames that are printed by the local command. These should be recognized as local addresses. Try some other hostnames, such as 'example.com'. These should be recognized as non-local addresses. If either one is not the case, your code is buggy. Go back and fix it, compile, create a new JAR, and test again; until it works as expected.

When the command is working as expected, play around some more. Can you find local hostnames that are not printed by the local command? For example, try your computer's name without a domain, or with domain '.local'. What about other IP addresses, like '127.0.0.8'? It's OK to play for a while, you've earned it!

Fix the lookup Command

Call the lookup command of your JAR. Interpret the stack trace and locate where the code is missing. Take in the surroundings. What input data do you have? What should the missing code do? There are descriptions and hints in the source code. What data will you need?

The comments point to a helper method you can use. Read the description of the helper. What arguments does it expect? What output does it provide? Don't bother with its implementation now, stay focused on the problem. Search for places where the helper is already used, to see how it is done there.

The helper expects objects of class InetAddress. From what package is that class? Browse its JavaDocs. Any hints on how to obtain such objects? From a hostname, which is the only input you have? There are different methods you could call, which is the right one? If you're not sure, go back to where the code is missing, read the description and comments again.

OK, you've got all the pieces of the puzzle now. It is time to write code. Remove calls to class Missing. Put in code that looks up the internet addresses for the hostname. Implement a loop that prints them, using the helper method. Is there a problem when compiling? Handle the exception that might be thrown by the lookup. The user should know when that fails.

Create a JAR and test your brand new implementation of lookup. Below are some hostnames to try. Most of them require an internet connection for the lookup.

  • localhost  -  a subset of the output from the local command
  • nowhere.invalid  -  no address at all
  • gnu.org  -  a single IPv4 address
  • apache.org  -  two different IPv4 addresses
  • fsf.org  -  both an IPv4 and an IPv6 address

Perform some more lookups of your own. If everything works, go back to your code once more. Read it again. Then clean it up. Is the code properly formatted? Are there leftover comments from before, which no longer make sense? Will somebody else understand what you've written, or do you have to add comments of your own? In the latter case, consider rewriting your code so that it can be understood without comments.

Congratulations, you fixed the lookup command all by yourself. Good job!

Clean Up

The class Missing is just an artifice to mark the places where you have to insert code. That's why it does not appear in the generated JavaDocs for this tutorial. Now that you have fixed both of the broken commands, Missing is no longer needed.

Delete both the source file and the compiled class from your project. If you are using an editor rather than an IDE, delete the directory 'pityoulish/outtake/' and everything within. Then re-compile the classes in package pityoulish.tutorial and create a new JAR. There might be an unused import statement that you have to remove.

Splendid! You have successfully completed this tutorial.

Recipe

The Java classroom exercises adhere to a common pattern. The following recipe for solving them comprises the development procedures established above. After completing this tutorial, you are well prepared for the classroom.

Understand & Plan

Read the exercise description. Download the JAR and read the usage instructions. There is a set of commands with a common topic. Unlike in the tutorial, those commands are not independent. Some of them require the output of an other as input. Prepare a sequence in which you can execute the commands. Think of it as a user session, a typical scenario for using the JAR. If necessary, prepare different sequences for different groups of commands. The exercise description might help you with that.

Study the JavaDocs provided with the exercise. Can you spot the classes which implement the commands? Which packages and classes contain the 'real' functionality? Which contain only utility stuff and supporting functions? Don't dive into the source code yet, just get an overview.

Give the JAR a decent name. Create a project in your IDE or simply extract the JAR.

Test & Fix

Execute your command sequence until you encounter a Missing exception. Interpret the stack trace and find where the code is missing. Take in the surroundings. What should the missing code do? Should it print something, initialize attributes, perform IO? What input data do you have available, from arguments or stored in attributes?

What APIs can you use to solve the problem? What APIs are already in use? Read their JavaDocs. Does the standard Java API provide what you need? Are there helpers in the code of the programming exercise? Are there other commands that do something similar? Can you re-use ideas or code snippets from those?

Insert the missing code. Compile it and create a new JAR. Test the changes. Does the command now work as expected? Consider error cases, too. If the command works, clean up your code before you move on. The instructor might come around and ask to see it.

Continue with the command sequence until you can execute it all the way through. When you're done with all the command sequences, delete class Missing. This is the ultimate check that you didn't overlook something.
